# DevForge Design Context

## Design Context

### Users
- **目标用户**：通用 IT 人员 —— 开发者、运维工程师、DBA、测试人员
- **使用场景**：日常开发运维中需要管理数据库（MySQL/PostgreSQL/SQLite）、SSH 远程连接服务器、SFTP 文件传输、批量执行命令
- **核心诉求**：一个工具替代 Navicat + DataGrip + FileZilla + 终端模拟器，减少工具切换，提高效率
- **使用频率**：工作日全天高频使用，长时间驻留桌面

### Brand Personality
- **品牌个性**：专业 · 高效 · 精致
- **语气**：沉稳可靠但不刻板，现代新颖但不花哨
- **情感目标**：让用户感到 "这个工具懂我" —— 安心、掌控感、专注
- **名称含义**：DevForge = Developer's Forge（开发者的锻造炉），暗示打磨、精工细作

### Aesthetic Direction
- **视觉调性**：JetBrains / DataGrip 风格 —— 信息密度高、功能分区清晰、深色主题为主、专业工具感
- **主题**：暗色优先（dark-first），亮色作为可选方案
- **配色基底**：zinc 色系（oklch），已有完整 design tokens，蓝紫色调的微妙色相偏移（hue ~286）
- **字体**：Inter + 系统回退链（适合高密度信息展示）
- **图标**：Lucide（线条风格，统一 stroke width）
- **组件库**：shadcn/vue (new-york style) + reka-ui 底层
- **编辑器**：Monaco Editor（SQL/代码）、xterm.js（终端）
- **反面参考（NOT like）**：
  - 不要像消费级 SaaS 产品（大留白、圆润卡片、渐变英雄区）
  - 不要像 Electron 套壳网页（迟钝、臃肿）
  - 不要 AI 味设计（紫蓝渐变、霓虹暗色、玻璃态到处用）

### Design Principles

1. **信息密度优先（Density Over Decoration）**
   专业工具用户需要在一屏内看到尽可能多的有效信息。宁可紧凑也不要为了 "好看" 大量留白。每一像素都应该为信息服务。

2. **操作零等待（Instant Feedback）**
   所有交互必须有即时反馈。数据库查询结果流式加载、文件传输实时进度、SSH 命令零延迟回显。不让用户等、不让用户猜。

3. **键盘优先（Keyboard-First）**
   目标用户大量使用键盘。快捷键覆盖所有高频操作，Tab 导航逻辑清晰，搜索/命令面板随时可达。鼠标只是辅助。

4. **沉稳不花哨（Calm Confidence）**
   动效克制、有目的。不用纯粹装饰性动画，只用功能性动效（状态变化、面板展开、结果加载）。颜色以中性色为主，彩色仅用于语义（成功/警告/错误/信息）。

   **允许且鼓励的装饰元素**：在合适的场景下可以添加精心设计的装饰细节来提升高级感和品牌辨识度，这是 DevForge 区别于传统工具的核心特色：
   - `industrial-grid` 网点背景 — 用于面板/区域的工业质感层，保持低 opacity
   - `noise-texture` 噪点纹理 — 为表面添加手工质感，避免"塑料感"
   - 状态指示器动画（`animate-pulse` 小圆点）— 用于连接状态、就绪状态等有语义的场景
   - 选中态装饰（如音频条形态指示器）— 用于增强当前选中项的视觉反馈
   - 按钮 hover 微交互（图标旋转、grid overlay 显现）— 增加操作反馈的精致度
   
   **关键原则**：装饰元素必须服从整体信息层级，不能喧宾夺主。opacity 控制在 5%-20% 之间，确保可读性优先。

5. **一致到像素（Pixel-Perfect Consistency）**
   间距、圆角、阴影、字号、图标尺寸全部走 design tokens。不同模块（数据库/终端/SFTP）保持统一的视觉语言，切换模块时不应感到 "换了个 app"。
